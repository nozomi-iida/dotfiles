# nozo-blog 記事作成コマンド

## 概要

nozo-blogリポジトリに記事(mdx)を作成・更新する。新規記事は原則 `/plan-nozo-article` で決めた方向性（プラン）を読み取って本文化する。
どのClaude Codeセッションからでも実行できるよう、nozo-blogへは**絶対パス**で書き込む。

> 役割分担: **方向性を決める** のは `/plan-nozo-article`、**本文を書く** のがこのコマンド。

**nozo-blogのパス**: `/home/nozomi/ghq/github.com/nozomi-iida/nozo-blog`
**記事の保存先**: `{nozo-blog}/src/articles/{category}/{YYYY_MM_DD}_{slug}.mdx`

## 引数フォーマット

```
$ARGUMENTS = [--category programmings|poems|atcoder] [--title タイトル] [--slug slug] [記事にする内容の補足・テーマ]
```

### オプション引数

- `--category <name>`: カテゴリ（未指定時は `programmings`）
- `--title <title>`: 記事タイトル（未指定時は会話内容から自動生成）
- `--slug <slug>`: ファイル名用の英語スラッグ（未指定時は会話内容から自動生成）
- その他: どの話題を記事にするか・補足したい内容（会話が長い場合の絞り込みに使う）

## 実行手順

### Step 0: セッション初回の方針確認（初回のみ）

**このセッションで本コマンドを実行するのが初めての場合**、いきなり書き始めず、まず `AskUserQuestion` で「何をするか」を確認する。

- **既存の記事を更新する** → 「分岐A: 既存記事の更新」へ
- **新しく記事を作る** → 「分岐B: 新規記事の作成」へ

**2回目以降の実行では、この確認はスキップする。** 同一セッションで既に選んだ方針・対象記事の続き（追記・修正）とみなして進める。ただしユーザーが明示的に別のこと（「別の記事を作りたい」「今度は更新したい」等）を指示した場合はその指示を優先する。

> 補足: `$ARGUMENTS` に更新/新規や対象が明確に書かれている場合（例: 既存記事のslug指定、`新しく〜の記事を書いて` 等）は、それを方針として採用し、Step 0 の確認は省略してよい。

---

### 分岐A: 既存記事の更新

#### A-1. 更新対象の選択

記事一覧を提示し、どの記事を更新するかをユーザーに確認する。

```bash
ls -t /home/nozomi/ghq/github.com/nozomi-iida/nozo-blog/src/articles/*/*.mdx
```

- `$ARGUMENTS` で対象（slug やタイトルの一部）が指定されていれば、それで絞り込む
- 候補が複数あるときは `AskUserQuestion` で選んでもらう

#### A-2. 現状の読み込みと更新方針のヒアリング

対象記事を `Read` で読み込み、現在の内容を把握したうえで、どう更新したいかをユーザーに確認する（例: セクション追加、内容の修正、続編相当の追記など）。会話の内容から更新点が明確な場合は、その要約を提示して方針として確認する。

#### A-3. 更新の適用

`Edit` で対象箇所を書き換える。スタイルガイド（後述）に従う。`publishedAt` は原則変更しない（公開日を保つ）。`isPublished` は勝手に変更しない。

#### A-4. 結果の表示

更新したファイルの絶対パスと、変更した箇所の要約を表示する。Step 5 の案内（ローカル確認方法など）も行う。

---

### 分岐B: 新規記事の作成

新規記事は、原則 `/plan-nozo-article` が作った**骨組みmdx（タイトル・目次だけで本文が空のファイル）の目次を埋める**。タイトル・slug・目次・ファイル自体は `/plan-nozo-article` が既に作っているので、このコマンドは本文執筆に専念する。以下の Step 1 以降に従う。

### Step 1: 引数の解析

`$ARGUMENTS` から `--category` / `--slug` と、その他の補足指示を抽出する。

### Step 2: nozo-blogの存在確認

```bash
test -d /home/nozomi/ghq/github.com/nozomi-iida/nozo-blog/src/articles && echo OK
```

存在しない場合はエラーを表示して終了する。

### Step 3: 骨組みmdxの特定

`/plan-nozo-article` が作った骨組みmdx（フロントマター + `##` 見出し + `{/* メモ */}` だけで、本文がまだ書かれていないファイル）を探す。

```bash
ls -t /home/nozomi/ghq/github.com/nozomi-iida/nozo-blog/src/articles/*/*.mdx
```

- `$ARGUMENTS` で slug が指定されていれば、それで絞り込む
- 候補が複数あるときは `AskUserQuestion` で対象を選んでもらう
- 対象を `Read` で読み込み、タイトル・目次（見出し）・各見出しの `{/* メモ */}` を執筆の土台にする

**骨組みが見つからない場合の扱い:**

- まだ骨組みを作っていないなら、「先に `/plan-nozo-article` でタイトル・目次を決めることをおすすめします」と案内する
- ただしユーザーが会話の内容からそのまま書きたい旨を示している場合や、`$ARGUMENTS` に十分なテーマ指定がある場合は、骨組みなしでも執筆を進めてよい。その場合は自分でタイトル・slug・目次・`publishedAt` を決めてファイルを新規作成する（`date -u +"%Y-%m-%dT%H:%M:%S.000Z"` / `date +"%Y_%m_%d"`、カテゴリ未指定なら `programmings`、slug重複は要確認）

### Step 4: 本文の執筆

骨組みの各見出しの下に、`{/* メモ */}` と会話内容をもとに本文を書いていく。下記のフォーマット・スタイルガイドに従い、`Edit` で骨組みを埋める形で書く。

- 目次（見出し構成）は原則維持する。書いてみて過不足があれば見出しの追加・調整をしてよいが、大きく変える場合はユーザーに確認する
- 執筆後、`{/* メモ */}` などのプレースホルダーコメントは削除する
- フロントマター（`title` / `publishedAt` / `isPublished`）は骨組みのものを維持する（`isPublished` は勝手に変えない）

### Step 5: 結果の表示

- 書き上げたファイルの絶対パスを表示
- `isPublished: false`（下書き）のままであることを通知
- 「内容を確認し、公開する場合は `isPublished: true` に変更してください」と案内
- ローカル確認方法として `npm run dev`（nozo-blogディレクトリで `-p 3333`）を案内

## 記事のフォーマット

### フロントマター

```mdx
---
title: 記事タイトル（日本語、簡潔に）
isPublished: false
publishedAt: {ISO 8601形式の日時。骨組みがあればそれを維持、新規作成時は date -u +"%Y-%m-%dT%H:%M:%S.000Z" で取得}
---
```

**isPublished はデフォルトで `false`（下書き）**。

### 構造

1. **導入文**（1-3文）
   - なぜこの記事を書くのか、背景や動機
   - 「最近〜」「今回は〜」などで始める
2. **本文**
   - `##` で大きなセクション、`###` で小さなサブセクション
   - 必要に応じて箇条書きを活用
3. **コードブロック**（必要な場合）
   - 言語を必ず指定（```typescript、```bash など）
   - コードの後に何をしているか日本語で説明を追加
4. **まとめ/感想**（任意）
   - 学んだことや今後の展望
5. **参考資料**（任意）
   - 参考にしたリンクをマークダウンリンク形式でまとめる

## スタイルガイド

### 文体

- ですます調を使用
- 一人称は「自分」
- 実践ベースの表現（「〜してみました」「〜を試した」）
- 感想は柔らかい表現（「〜と思う」「〜なと感じた」）

### コードの説明

- コードブロックの後に、何をしているか日本語で説明を追加
- 重要なポイントや注意点があれば補足

### リンク

- 公式ドキュメントへのリンクは積極的に挿入
- URLだけでなくマークダウンリンク形式を使用

### 避けるべきこと

- 長すぎる導入文
- 抽象的な説明のみ（具体例を入れる）
- 会話のタイムスタンプやセッション固有の情報をそのまま残す

## ファイル命名規則

`{YYYY_MM_DD}_{slug}.mdx`

- 日付: コマンド実行時の日付（`date +"%Y_%m_%d"`）
- slug: 会話内容から考えた英語のケバブケース

### slugの決め方

記事の主題を最も適切に表す英語slugを考える。

**ルール**:

- ケバブケース（小文字、ハイフン区切り）
- 簡潔で内容がわかるものにする（長すぎない）
- 技術名やツール名はそのまま使用（例: `inversify`, `vitest`, `chakra-ui`, `nix`）

**パターン**:

- How-to系: `how-to-{動詞}-{対象}`（例: `how-to-use-inversify`）
- 設定系: `setup-{対象}`（例: `setup-logger-express`, `setup-ogp`）
- 移行系: `migrate-{from}-to-{to}` / `migrate-{対象}`（例: `migrate-chakra-ui-v3`）
- 学習系: `about-{対象}` / `learning-{対象}`
- 紹介系: `{ツール名}-{機能}`（例: `tanstack-table`, `prisma-seed`）
- 管理系: `manage-{対象}-with-{ツール}`（例: `manage-cli-tools-with-nix`）

## 重要な注意事項

1. **セッション初回は方針を確認する**: 初回実行時は「既存記事の更新」か「新規作成」かを確認する（Step 0）。2回目以降は直前の続きとみなして確認を省略する
2. **方向性のヒアリングはしない**: 新規記事のテーマ・構成決めは `/plan-nozo-article` の担当。このコマンドはプラン（または会話・引数）を土台に本文を書くことに専念する
3. **絶対パスで書き込む**: nozo-blog以外のディレクトリから実行されても動くよう、保存先は常に絶対パス
4. **下書きで作成**: `isPublished: false` をデフォルトとし、勝手に公開しない
5. **git操作はしない**: コミットやプッシュはユーザーに任せる
6. **会話固有情報の除去**: セッション固有のパスやタイムスタンプを記事に残さない
7. **日本語**: タイトル・本文は日本語で記述

## 使用例

```bash
# 直近のプラン(/plan-nozo-articleで作成)を読み取って本文化
/write-nozo-article

# 特定のプラン(slug指定)を本文化
/write-nozo-article --slug how-to-use-herdr

# プランなしで、会話の内容からそのまま記事化
/write-nozo-article Nixでdotfilesを管理した話
```
